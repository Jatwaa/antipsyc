The problem

A model that says "the file exists" is making a claim — not reporting an observation.

Large models — especially smaller, faster ones — routinely assert things that are not true. They confuse memory with observation, simulation with measurement, and fluency with fact.

  • A model that "remembers" a file existed may not know it was deleted.
  • A model that "knows" the API returns 200 hasn't checked since the service went down.
  • A model that says the tests pass may be extrapolating from patterns, not running them.

The failure mode isn't lying. It's not knowing the difference between believing and knowing. AntiPsyc makes that difference structural.

How it works

The claim lifecycle

Every statement is a hypothesis until a real tool — reading the filesystem, making an HTTP request, running code — produces a matching result. Only then does it earn a confidence score and permission to be presented to a user.

C Claims

Statements submitted by a model. Each gets a type, a status, a confidence score, and a realityWeight — how well-grounded in external reality it currently is. Claims start at 0.1 and only rise through verified evidence.

E Evidence

Immutable, append-only records. Every validator run appends a new record; nothing is ever deleted. The full chain shows exactly how a claim's confidence evolved and which run drove each change.

G The gate

Translates a raw realityWeight into a presentability signal. A contradicted claim always suppresses. Without explicit verified/contradicted flags, the gate defaults to suppress rather than assuming optimism.

The confidence gate

One signal that tells the model how to speak

After verification, the gate maps the grounded realityWeight to exactly one of three instructions. Since v0.7 it's embedded in every verification response — verdict and presentation guidance in a single round trip.

verified
rw ≥ 0.75
Strong external grounding. Assert the claim directly.
caveat
rw ≥ 0.40
Partial verification. Qualify before asserting — "based on available evidence…".
suppress
otherwise / contradicted
Could not be verified, or was found false. Disclaim or omit — do not assert.

The gate can read the evidence ledger itself by claimId — so a hallucinating model cannot fabricate a high score and bless its own claim.

Grounded validators

The model proposes. Validators confirm.

Each validator is a real check against external state, and the claim type constrains which validators may be used — you can't verify a filesystem claim with an HTTP check, or an HTTP claim with self-supplied text.

filesystem.exists filesystem.stat file.contains file.matches codebase.contains json.valid json.path math.evaluate code.run process.run http.fetch retrieve_and_ground git.file_exists git.contains git.branch_exists git.log_contains git.last_modified git.blame_line text.contains interaction.chain http.json_path file.hash symbol.exists glob.count

Scope must match

A validator that confirms a file exists cannot confirm the system is "healthy" or "secure". Broad claims against narrow evidence are demoted, not promoted.

Evidence decays

The world changes. Each validator has a TTL; stale evidence is disclaimed, not silently trusted. Re-running the validator refreshes it.

Contradictions are permanent

A claim that has been contradicted carries that record forever. The full history is always available to any caller.

Adversarial scenarios — attempt → outcome

A model supplies its own "evidence" text to verify a real-world fact→syntactic · rw ≤ 0.25
Unrelated math used to "prove" a broad operational claim→demoted · irrelevant
A high realityWeight claim that evidence contradicted→gate · suppress
A claim padded with clauses the validator cannot speak to→contract · rejected
A fabricated realityWeight handed straight to the gate→unattested · suppress
A model confirms "it's done / tests pass" on its own word→new gate · forced validation
Satisfying a forced gate with an unrelated true fact, or only half the named files→HALT · every artifact or none
Reading a file outside the configured allowed roots→blocked
An HTTP probe to loopback or a private IP (SSRF)→blocked
A public URL that redirects to a private address→blocked at the hop
An interaction chain asserting a physical event with no causal schema→cannot verify

Get started

Running in under a minute

Requires Node.js 18+ (22+ enables SQLite persistence). No install step, no dependencies to download.

Run the server (HTTP + MCP) 
# clone and run — that's it
git clone https://github.com/Jatwaa/antipsyc
cd antipsyc
node src/server.js --http --mcp

# evidence UI → http://127.0.0.1:8717
Register with an MCP client 
# Claude Code
claude mcp add antipsyc -- \
  node ./src/server.js --mcp

# then, in your agent:
# 1. verify_claim → returns evidence + gate
# 2. assert only when gate = "verified"
Verify a claim over HTTP 
POST /api/verify
{
  "statement": "package.json exists",
  "validator": "filesystem.exists",
  "path": "package.json"
}
// → { verified, realityWeight, gate, … }

Design principles

What AntiPsyc believes

Assertion is not evidence

A model saying something confidently is a prior, not a signal about truth. Validator results are posteriors. The gate uses posteriors.

Grounded or it doesn't count

The validator — not the model — reads the file, makes the request, runs the command. The model supplies the claim; the tool supplies the observation.

Scope must match

The claim must be as narrow as the evidence. A healthcheck file existing does not prove the system is healthy.

Evidence decays

A fact that was true 20 minutes ago may not be true now. Stale evidence is disclaimed, not silently cached.

AntiPsyc
Claims remain provisional until tools produce evidence.

AntiPsyc is a verification and presentation-guidance layer. Its gate signals are advisory — enforcement happens in the consuming agent. It reduces hallucinated assertions about checkable facts; it is not a guarantee of correctness.
